# Reado — Design Guidelines

> Persistent design context for all sessions. Grounded in the OpenSpec change `add-reado-mvp` and the verified research in `docs/research/color-theory-for-reading.md`. Reado is an open-source, read-first code IDE. All UI copy in English (i18n: IT + EN).

## Design Context

### Users
Experienced software engineers (the author first; open-source devs later) reading code they did not write — their own legacy, teammates', or unfamiliar codebases — to *understand* it. Context: long, focused sessions; the user is the reviewer, the AI agent is the committer. They open Reado *instead of* their write-first editor specifically to read calmly and annotate.

### Brand Personality
Three words: **calm, precise, trustworthy.** Voice is quiet and confident — never loud, never playful-for-its-own-sake. The interface should evoke **focus and ease**, the feeling of a well-lit reading desk, not a cockpit. Trust is paramount: nothing changes the user's code without an explicit trigger, and the tool never hides state (orphaned anchors, agent activity, and diffs are always honest and visible).

### Aesthetic Direction
Minimal, editorial, content-first. The **code is the hero**; chrome recedes. References in spirit: the restraint of Solarized and iA Writer's reading focus; the structural clarity of Linear. Anti-references: the "Christmas tree" rainbow theme, maximal-contrast pure-black-on-white, busy multi-panel cockpits, and anything that reads as a generic AI-generated dashboard (no gratuitous gradients, no decorative glows). Every element earns its place by function or real visual impact — otherwise it is removed.

### Design Principles
1. **Read-first.** Default view is clean, syntax-highlighted code — no diff, no AI overlay. All reading aids (focus mode, comfortable measure, soft landing highlight) serve sustained comprehension.
2. **The comment is the unit.** Annotation is the primary action; it is an external overlay that never modifies the code. Markers are quiet until engaged.
3. **Explicit triggers, never silent action.** The AI acts only on an explicit command. State is never faked — a failed anchor becomes a visible orphan, never a wrong line.
4. **Restraint over decoration.** Few semantic colors, generous whitespace, subtle motion. When in doubt, remove.
5. **Honest, accessible surfaces.** WCAG AA is the floor; colorblind-safe; reduced-motion respected; the same semantic role keeps the same perceptual position across themes.

## Color System

Derived from the research (`docs/research/color-theory-for-reading.md`). Key evidence: highlighting has **no proven comprehension-accuracy benefit** and helps experienced readers little → keep it **subtle**; never pure-black-on-white; build in **OKLCH**; ship **both** light and dark; anchor the limited palette on **control-flow first**, then definitions, then strings.

### Methodology (how every theme is built)
- Author all colors in **OKLCH** (perceptually uniform, CSS-native). Derive sRGB from it.
- **≤6 semantic syntax colors** over base + text. Accents share near-equal lightness/chroma; hue varies by fixed angular steps (colorwheel relationships).
- **Bases are desaturated, contrast comfortable — not maximal.** Dark base is gray-blue, never `#000`. Light base is warm off-white, never `#fff`.
- **Same semantic role = same perceptual position across themes** (hold hue/role constant), but **tune lightness/chroma per mode** to hit WCAG AA in *both* (Solarized's mistake was reusing identical accents → AA failures in light mode).
- **Colorblind-safe:** separate accents on the **lightness axis as well as hue**; verify in a CVD simulator (deuteranopia/protanopia/tritanopia). Where two roles risk confusion (control-flow vs definition), add a redundant non-color cue (weight/italic).
- **Comment-annotation markers live OUTSIDE the syntax palette** — a distinct hue reserved for Reado's own overlay, so they never read as code.
- Verify text/background pairs with **WCAG AA (floor) + APCA (fine-tuning)**.

### Semantic roles (priority order from eye-tracking research)
1. Control-flow keywords (`if`/`for`/`return`) — most salient
2. Other keywords / storage (`const`/`fn`/`class`)
3. Definitions (function & type names)
4. Strings
5. Numbers / constants
6. Comments — deliberately **de-emphasized** (low chroma)

Punctuation/operators stay near the base text color (muted).

## Typography & Reading
- High-legibility monospace for code, **configurable**; final default chosen on-screen during the impeccable build pass (candidates: JetBrains Mono, Commit Mono, Berkeley Mono). Must disambiguate `0/O`, `1/l/I`.
- Comfortable reading measure for long-form reading; line-height tuned for sustained reading, not density.
- Line wrap **off** by default (quick toggle).

## Motion
- Subtle and purposeful only (e.g., soft landing-line highlight that fades, popover entrance). Respect `prefers-reduced-motion`. No decorative ambient animation.

## Theme selection (resolved)
Three modes: **Manual**, **System** (follows OS, the default mode), and **"Trust Reado"** — a time-of-day auto mode that picks light by day and dark in the evening/night (by local sunrise/sunset when available, else configurable times), switching live. Both light and dark are first-class; neither is imposed. This matches the research: polarity advantage is conditional, so let context decide.
